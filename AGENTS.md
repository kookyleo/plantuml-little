# plantuml-little — Agent Guidelines

## Project Identity

**plantuml-little** is a lightweight Rust port of PlantUML, targeting a single use case: `.puml` → `.svg` conversion, delivered as a library + CLI tool.

**Current phase**: Upstream alignment — SVG output must be byte-identical to Java PlantUML.

## Hard Boundaries

### In Scope
- Parse PlantUML text (`.puml` files)
- Render to SVG only
- Library crate (`plantuml-little`) + binary crate
- Graphviz layout via system `dot` command (`dot -Tsvg`, parsed for coordinates)
- Self-contained layout for other diagram types

### Out of Scope — Do NOT implement
- Any output format other than SVG (no PNG, PDF, EPS, HTML5, ASCII, etc.)
- GUI, web server (Picoweb), FTP server, pipe mode
- PlantUML Server URL encoding/transcoding
- Security sandbox / profile system
- ELK layout engine
- TeaVM / JS compilation
- PNG metadata embedding/extraction

## Architecture

```
puml text → preprocess → parse → layout → render → SVG string
```

## Maintenance Guidelines

4. **stdlib updates** -- vendor content updates as needed

### Test Strategy

Three-tier testing:

1. **Unit tests** (1321): within each parser/layout/render module
2. **Integration tests** (183): `.puml` → `.svg`, verify valid SVG + no Creole markup leakage
3. **Reference tests** (296): byte-for-byte comparison against Java PlantUML output
   - Reference SVGs in `tests/reference/`, generated from Java PlantUML 1.2026.3beta4
   - Regenerate: `bash tests/generate_reference.sh`
   - Test macro: `reference_test!` in `tests/reference_tests.rs`
   - Generated by: `python3 tests/generate_test_list.py > tests/reference_tests.rs`

Font metrics extracted from Java AWT via `tests/tools/ExtractFontMetrics.java` → `src/font_metrics.rs`

### Upstream Alignment — The Iron Rule

**目标：SVG 输出与 Java PlantUML 逐字节一致。**

#### 最高原则

**不要凑数拟合。读 Java 源码，使用完全相同的算法和参数。不要有丝毫侥幸。**

一个 bit 的差异都会让 reference test 失效。任何"差不多"的近似都不可接受。

#### 实践方法论

1. **读源码优先**。当输出有差异时，找到 Java 中对应的代码路径，理解其精确算法。关键文件：
   - `SvgGraphics.java` — SVG 元素生成、坐标格式化 (`format()`)、`ensureVisible()` 机制
   - `SvekResult.java` — `calculateDimension()`、`moveDelta(6, 6)`
   - `DotStringFactory.java` — Graphviz 参数 (`nodesep`, `ranksep`, `minRankSep`)
   - `SvekNode.java` — 节点位置提取 (`moveDelta`、`getMinX/getMinY`)
   - `LimitFinder.java` — bounding box 计算（`drawRectangle` 加 `x-1,y-1`、`UEmpty` 修正）
   - `FontParam.java` — 每种元素的默认字号
   - `SkinParam.java` — 圆角字符半径等默认值
   - `EntityImageClass.java` / `EntityImageClassHeader.java` — 类图内部布局
   - `HeaderLayout.java` — 圆角字符 + 名称 + 泛型的排列
   - `TextBlockExporter12026.java` — `calculateFinalDimension()`、document margin (R=5, B=5)

2. **在 Java 代码中插入调试输出**来获取精确的中间值。例：
   ```java
   // SvekResult.java — 查看 LimitFinder 计算结果
   System.err.println("DEBUG_MINMAX: minX=" + minMax.getMinX() + " maxX=" + minMax.getMaxX());

   // MinMaxMutable.java — 追踪 bounding box 变化来源
   if (y > maxY) System.err.println("DEBUG: maxY -> " + y + " from " + new Throwable().getStackTrace()[2]);

   // SvgGraphics.java — 追踪 viewBox 最终值来源
   // ensureVisible 中打印 maxX/maxY 变化和调用栈
   ```

3. **逆向验证**。用已知正确的 Java 输出逆推公式，再用 Java 代码确认：
   - 从 reference SVG 提取坐标值
   - 计算：reference_value = f(哪些参数？)
   - 在 Java 源码中找到 f 的精确定义
   - 在 Rust 中实现相同的 f

4. **不要猜测常量**。以下常量必须从 Java 源码中读取，不得拟合：
   - `MARGIN = 6` (SvekResult moveDelta 偏移)
   - `CANVAS_DELTA = 15` (SvekResult delta(15,15))
   - `DOC_MARGIN_RIGHT = 5, DOC_MARGIN_BOTTOM = 5` (plantuml.skin document style)
   - `nodesep = max(0.35, 35/72)`, `ranksep = max(0.8, 60/72)` (DotStringFactory)
   - `HEADER_HEIGHT = 32`, `EMPTY_COMPARTMENT = 8` (从 reference 验证 + Java 代码确认)
   - `CIRCLE_LEFT_PAD = 4`, `CIRCLE_DIAMETER = 22`, `circle_text_gap = 3`, `RIGHT_PAD = 3`

5. **Graphviz 集成必须与 Java 完全一致**：
   - 使用 `dot -Tsvg`（不是 `dot -Tplain`，plain 格式有 inches→pt 精度损失）
   - 从 SVG polygon 读取节点 min(x), min(y) 作为位置（Java 的 `SvekUtils.getMinXY`）
   - 保留原始 edge path d-string（M/C/L 命令）
   - 保留原始 arrowhead polygon 坐标
   - 节点尺寸用自己精确计算的值，不用 Graphviz 的四舍五入值

6. **字体度量必须精确**：
   - 从 Java AWT `FontMetrics.getStringBounds()` 导出精确查找表
   - 使用 `:.6f` 格式（6 位小数），不是 `:.6`（6 位有效数字）
   - 工具链：`ExtractFontMetrics.java` → `font_metrics.json` → `gen_metrics_table.py` → `font_metrics.rs`

#### 调试工作流

```
1. 运行 reference test，定位第一个差异字节
2. 确定差异属于哪个类别（坐标/属性/结构/内容）
3. 找到 Java 中生成该值的代码路径
4. 在 Java 代码中加 debug 输出，重新编译运行，获取精确中间值
5. 在 Rust 中实现相同算法
6. 验证：cargo test --test reference_tests
7. 提交
```

Java PlantUML 源码位置：`/ext/plantuml/plantuml`
构建：`cd /ext/plantuml/plantuml && ./gradlew jar`
运行：`java -jar build/libs/plantuml-*.jar -tsvg -pipe < input.puml 2>debug.txt > output.svg`

### Parallel Agent Work

File ownership matrix (enforced during parallel agent work):
- Each agent only modifies its assigned files
- parser/layout/render naturally isolated by diagram type
- Shared files (lib.rs, mod.rs, diagram.rs) modified sequentially by the main thread
